# Pattern Catalog — SOLID Principles and Design Patterns (Java)

This is the course's living cheat-sheet: one entry per principle and per pattern, added as we learn it. It is what you reread before any design decision or interview.

**How to read an entry.** Each one answers the same questions in the same order, so you can scan them side by side:

- **Name & intent** — what it is, in one plain sentence.
- **The pain it cures** — the specific problem that makes you reach for it.
- **Use it when…** — the concrete signals that it fits.
- **Do NOT use it when…** — the equally concrete signals that it is the wrong tool. (Over-applying patterns is the classic beginner mistake; this field is never left thin.)
- **The cost** — the indirection or complexity it always adds. Nothing is free.
- **Relatives** — patterns it resembles, and the one-line difference.
- **Seen in the wild** — one place the JDK or a popular framework already uses it.
- **Where it lives** — its kata package, and its home in the Smart Home Hub if it has one.

The golden rule sits above every entry: **a pattern is a named cure for a specific pain.** If you cannot name the pain, you do not yet need the pattern.
